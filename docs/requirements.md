## 1. プロジェクト概要

### 1.1 背景・目的

- **背景**: 冷蔵庫内の食材や期限の把握が難しく、廃棄や買い忘れが起きやすい。
- **目的**: 冷蔵庫の中身・期限・残量を記録し、買い物や調理の判断をしやすくする。

### 1.2 スコープ

| 対象 | 内容 |
|------|------|
| **含む（In）** | 食材の登録・一覧、賞味期限・消費期限・残量の記録、（将来）レシピ、買い物リスト、端末間でのデータ参照 |
| **含まない（Out）** | 初版でのログイン、SNS連携、課金、店舗価格比較、バーコード自動読取（将来検討可） |

### 1.3 想定ユーザー

| 項目 | 内容 |
|------|------|
| **主な利用者** | 現時点は本人のみ。将来は家族など複数人利用の可能性あり。 |
| **利用シーン** | 買い物後・料理前にスマホで確認、自宅PCで一覧を見る。 |
| **ITリテラシー** | 一般的なスマホアプリ操作ができれば利用可能な水準を想定。 |

## 2. 記録の対象

### 2.1 ドメイン一覧

| ドメイン | 内容 | 初版（Must） |
|----------|------|----------------|
| **食材（冷蔵庫の中身）** | 名称、残量、賞味期限・消費期限など | **対象** |
| **レシピ** | 料理名、材料、手順など | 将来（Should 以降） |
| **買うもの** | 買い物リスト | 将来（Should 以降） |

**初版の対象**: **食材のみ**（Q7 確定）

### 2.2 食材1件に含める情報

| 項目 | 現行（Phase A） | 次フェーズ（Phase B） | 説明 |
|------|----------------|----------------------|------|
| 食材名 | 必須 | 必須 | 例：牛乳、にんじん |
| 残量 | 推奨 | 推奨 | 数値 + **単位（選択式）**（例：500 ml、2 個） |
| 賞味期限 | 任意 | **いずれか必須** ※1 | 品質保持期限 |
| 消費期限 | 任意 | **いずれか必須** ※1 | 安全上の期限（ある場合） |
| 保管場所 | 任意 | 任意 | 冷蔵 / 冷凍 / 野菜室 など |
| 購入場所 | 任意 | 任意 | 詳細追加のみ |
| 購入金額 | 任意 | 任意 | 詳細追加のみ |
| メモ | 任意 | 任意 | 開封日、ブランドなど |

※1 **次フェーズのルール**: 賞味期限・消費期限の **どちらか一方以上** を入力必須とする。両方の同時入力は引き続き不可（現行の排他入力を継承）。

### 3.5 追加UIの方針（2系統）

食材の登録は **簡易追加** と **詳細追加** の2通りとする。

| 経路 | トリガー | UI | 想定シーン |
|------|----------|-----|------------|
| **簡易追加** | 一覧右下の FAB（＋アイコン） | **モーダル**（画面遷移なし） | 買い物直後など、名前だけ素早く入れたい |
| **詳細追加** | モーダル内リンク / 空状態など | **追加画面（S-02）** | 残量・両方の期限・保管場所・メモまで入れたい |

**FAB から追加画面へ直接遷移しない**（モーダルを開く）。

#### 簡易追加で入力する項目

| 項目 | 現行（Phase A） | 次フェーズ（Phase B） |
|------|----------------|----------------------|
| 食材名 | 必須 | 必須 |
| 期限（賞味または消費のいずれか1つ） | 任意 | **必須** |

※ 残量・単位・保管場所・購入場所・購入金額・メモは **詳細追加のみ**。

#### 詳細追加で入力する項目

仕様書の全項目。期限は現行どおり **賞味・消費の排他入力**。次フェーズでは **いずれか一方の期限入力を必須** とする。

### 3.6 次フェーズ（Phase B）：登録時の期限必須化

| 項目 | 内容 |
|------|------|
| **対象** | 簡易追加（モーダル）・詳細追加（画面）の **新規登録** |
| **必須条件** | `bestBefore`（賞味期限）または `useBy`（消費期限）の **少なくとも一方** が入力されていること |
| **排他** | 両方同時入力は不可（現行仕様を継承） |
| **目的** | 期限管理・期限順表示・期限切れ通知の前提データを揃える |
| **未入力時** | 保存不可。エラー例：「賞味期限または消費期限を入力してください」 |
| **既存データ** | Phase B 移行前に期限なしで登録済みの食材はそのまま表示可。編集時に期限入力を促すかは Phase B 設計時に決定（Q13） |

## 4. 非機能要件

| ID | 区分 | 要件 | 目標 |
|----|------|------|------|
| NF-01 | 利用環境 | PC・スマホのブラウザ | スマホ優先、レスポンシブ |
| NF-02 | UI | 見た目 | **シンプルな一覧**（カレンダー中心ではない） |
| NF-03 | 認証 | ログイン | **初版は不要** |
| NF-04 | データ同期 | 別端末から同じデータを見たい | **Phase B で実装**（初版は単一端末・ブラウザ内保存） |
| NF-05 | 言語 | UI | 日本語 |
| NF-06 | 性能 | 一覧 | 家庭利用の件数（おおむね数百件）で快適 |
| NF-07 | 拡張性 | 複数人利用 | 将来に備え、データ構造・同期方式を拡張可能に |

### 4.1 端末間同期について（重要）

ログインなしで「スマホとPCで同じデータ」を実現するには、**クラウド上の共有ストア**が必要です。

| フェーズ | 内容 | 備考 |
|----------|------|------|
| **Phase A（初版 Must）** | 食材の追加・一覧 | まず動くものを作る |
| **Phase B（次フェーズ）** | クラウド同期、**登録時の期限必須化** | ログインなし案：共有コード・URL等（Q10 で方式選定） |
| **Phase C（将来）** | ログイン・複数人 | 家族共有時 |

**合意（Q9）**: 別端末同期は初版の次フェーズで問題なし。Phase A は端末内保存のみ。

## 6. 成功基準（初版）

- [ ] FAB から簡易追加モーダルで食材を登録できる（名称は必須）
- [ ] 詳細追加画面で全項目を登録できる
- [ ] 登録した食材を一覧で見られる（シンプルなリスト）
- [ ] スマホ・PCのブラウザで表示が崩れない
- [ ] 同一端末で再訪問してもデータが残る
- [ ] （Phase B 実施時）別端末から同じデータを参照できる

**初版では含めない（合意済み）**: 編集・削除、レシピ、買い物リスト、ログイン
